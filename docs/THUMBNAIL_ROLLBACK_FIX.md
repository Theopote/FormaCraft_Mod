# 🔧 缩略图渲染回滚修复

## 🐛 问题描述

用户反馈修改后的缩略图出现严重问题：

1. **渲染完全不正确**
   - 超出缩略图区域
   - 样式不对
   - 方块之间关系不对
   - 贴图不对

2. **性能问题**
   - 设置锚点后电脑非常卡

## 🔍 问题诊断

### 性能问题根源

**修改前的代码**（导致卡顿）：

```java
// 每次渲染都要扫描整个 256×256 图像
for (int py = 0; py < srcHeight; py++) {  // 256 次
    for (int px = 0; px < srcWidth; px++) {  // 256 次
        int argb = image.getRGB(px, py);
        // 检测内容边界...
    }
}
// 总计：256 × 256 = 65,536 次 getRGB() 调用
```

**问题**：
- 每次 UI 渲染（每帧）都要执行这个扫描
- 游戏运行在 60 FPS，意味着每秒要扫描 **3,932,160 次像素**
- `getRGB()` 是相对昂贵的操作
- 导致严重的 FPS 下降和卡顿

### 渲染问题根源

**使用 `Math.max` 导致过度缩放**：

```java
// 错误的策略
double scale = Math.max(scaleX, scaleY) * 0.95;
```

**问题**：
- `Math.max` 会选择更大的缩放比例
- 如果内容是横向的（宽 > 高），会按宽度缩放，导致高度超出
- 如果内容是纵向的（高 > 宽），会按高度缩放，导致宽度超出
- 结果：**缩略图超出显示区域**

**示例**：
```
内容: 200×100 (宽>高)
区域: 200×200

scaleX = 200/200 = 1.0
scaleY = 200/100 = 2.0
scale = Math.max(1.0, 2.0) * 0.95 = 1.9

结果宽度: 200 × 1.9 = 380  ❌ 超出 200！
结果高度: 100 × 1.9 = 190  ✅
```

### 过度参数化

**修改前的 ComponentThumbnailGenerator**：

```java
// 过于激进的参数
double scale = (THUMB_SIZE * 0.85) / maxDim;  // 0.85 填充度
int blockPixelSize = Math.max(6, (int) Math.ceil(scale * 1.5));  // 1.5 倍

// 过度扩展顶点（导致视觉混乱）
int[] topX = {x, x + halfSize + 1, x, x - halfSize - 1};  // +1/-1
```

**问题**：
- 填充度 0.85 过高，留边距不足
- 方块扩展 +1/-1 像素会导致重叠
- 复杂的纹理模拟（3×3 棋盘格）在小尺寸下反而杂乱

## ✅ 解决方案

### 修复 1：简化 `ImageRenderer.renderCentered()`

**策略**：完全移除内容检测，直接 1:1 缩放

```java
public static void renderCentered(DrawContext ctx, BufferedImage image, 
                                 int x, int y, int areaWidth, int areaHeight) {
    if (image == null || areaWidth <= 0 || areaHeight <= 0) return;
    
    int srcWidth = image.getWidth();
    int srcHeight = image.getHeight();
    if (srcWidth <= 0 || srcHeight <= 0) return;
    
    // 直接 1:1 缩放到区域尺寸
    renderScaled(ctx, image, x, y, areaWidth, areaHeight);
}
```

**优势**：
- ✅ **零扫描开销** - 不再每帧扫描图像
- ✅ **性能极佳** - 直接调用已有的高效 `renderScaled()`
- ✅ **永不超出** - 严格按区域尺寸渲染
- ✅ **代码简洁** - 从 ~80 行减少到 ~10 行

### 修复 2：保守的 `ComponentThumbnailGenerator` 参数

**平衡的参数**：

```java
// 修复后
double scale = (THUMB_SIZE * 0.75) / maxDim;  // 0.75（平衡填充和边距）
int blockPixelSize = Math.max(5, (int) Math.ceil(scale * 1.3));  // 1.3倍

// 标准等轴测坐标（无扩展）
int[] topX = {x, x + halfSize, x, x - halfSize};  // 标准
```

**对比**：

| 参数 | 激进版本 | 保守版本 | 原因 |
|------|---------|---------|------|
| 填充比例 | 0.85 | **0.75** | 留足边距，避免贴边 |
| 方块尺寸系数 | 1.5 | **1.3** | 适中大小，避免过大 |
| 顶点扩展 | +1/-1 | **无** | 避免重叠和视觉混乱 |
| 纹理细节 | 3×3 网格 | **2 个点** | 简洁，避免小尺寸下杂乱 |

### 修复 3：恢复简洁的顶面纹理

```java
// 简洁的 2 点纹理（vs 复杂的 3×3 网格）
if (size >= 6) {
    g2d.setColor(applyBrightness(red, green, blue, alpha, topBrightness * 1.1f));
    for (int i = 0; i < 2; i++) {
        int tx = x - size/4 + i * size/2;
        int ty = y + size/4;
        g2d.fillRect(tx, ty, 1, 1);
    }
}
```

**优势**：
- 简洁清晰
- 计算开销小
- 视觉效果足够
